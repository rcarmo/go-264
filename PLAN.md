# go-264 plan

`go-264` remains decoder-first: make scalar decoding boringly correct, widen the conformance surface, and only then spend maintenance budget on assembly or an encoder.

The old plan had grown into a transcript of individual CABAC and Direct-mode investigations. Those details remain available in Git and the trace scripts; this file records the state we can reproduce and the work that still has a useful definition of done.

## Working rules

1. Ground syntax, prediction and filtering changes in H.264, FFmpeg or x264 rather than fixture-specific heuristics.
2. Keep scalar code as the reference for every SIMD or GPU path.
3. Treat exact samples as the correctness gate. PSNR is a localisation aid, not acceptance criteria.
4. Preserve coded dimensions during reconstruction and crop only at visible-output boundaries.
5. Profile before adding low-level code, and keep a fast path only when the benchmark moves enough to justify it.
6. Keep temporary FFmpeg patches, fixtures, YUV streams and traces under `/workspace/tmp`.

## Reproducible baseline

The current hard gate is the pinned 300-frame Big Buck Bunny Annex B stream generated by `scripts/bootstrap_fixtures.sh`:

```text
SHA-256 1305bc99a369721c46e35e3af8cc3e5f893f653eb6f472830bc70f6fcf3841ff
640x360 yuv420p, High profile, CABAC, three B-frames
```

Against single-threaded FFmpeg 7.1.3, the decoder matches all visible Y, U and V samples in display order with in-loop deblocking enabled. The matching no-deblock run is also exact. The CABAC first-divergence gate compares 2,100 macroblock events without a differing field.

`TestFFmpegReferenceParityBBB` makes that result durable: it verifies the fixture hash and oracle version, checks the frame count and display ordering, and reports the first differing frame, plane, macroblock and pixel.

This proves a substantial High-profile path. It does not prove every legal H.264 stream.

## Decoder coverage

The working decoder includes:

* Annex B scanning, emulation-prevention handling and guarded SPS/PPS parsing.
* I/P/B slice headers, POC/DPB bookkeeping, reference marking, P-slice list modification, parsed B-list operands and output reordering.
* CAVLC and CABAC macroblock/residual decoding, including 8x8 transforms and I_PCM reset handling.
* I4x4, I8x8, I16x16 and chroma intra prediction.
* P/B inter partitions, quarter-sample luma, chroma interpolation, spatial/temporal Direct mode and the weighted prediction used by the oracle stream.
* 4x4/8x8 transforms, residual addition and scalar in-loop luma/chroma deblocking.
* Defensive bounds handling around malformed readers, frame storage, coefficient buffers and reconstruction helpers.
* Syntax, motion, reconstruction, scalar/SIMD parity, fuzz and architecture build gates.

The decoder currently concentrates on 8-bit 4:2:0 progressive Annex B material. FMO reconstruction, interlaced/MBAFF streams, uncommon weighted B-prediction combinations and a broad conformance corpus remain open.

## Near-term decoder work

### Broaden conformance

Add small, pinned streams that exercise syntax outside the BBB gate without turning CI into a video archive. Useful candidates include multiple IDR GOPs, frame-number and POC wrap, explicit B weighting, long-term references, field-coded material and legal edge cropping.

Each fixture needs a recorded provenance, codec parameters and SHA-256. Acceptance is display-order Y/U/V equality against a pinned decoder, with a focused unit test for the primitive that failed.

### Retire assumptions exposed by new streams

The current oracle does not exercise the complete standard. New conformance streams should drive removal of assumptions around B-slice list modification, MaxPicNum values other than 16, weighted B prediction and field picture ordering -- one reproduced divergence at a time.

Avoid speculative rewrites. A theoretical gap becomes implementation work when a minimal fixture and a reference trace demonstrate it.

### Keep diagnostics useful

The CABAC, Direct-mode, BIDI and reconstruction scripts have earned their place because they turn a frame mismatch into a specific syntax, motion or pixel-stage discrepancy. They should remain opt-in, deterministic and quiet during ordinary decode.

Any new trace needs a comparator or a clear consumer. Hard-coded POC/macroblock probes belong in `/workspace/tmp`, not production code.

## SIMD and allocation work

Correct scalar output is the contract. Existing architecture wrappers and parity tests are the integration seam for measured optimisation.

The next pass should proceed in this order:

1. Re-profile the exact-parity tree on amd64 and arm64.
2. Measure batched IDCT/dequant and deblocking before writing assembly.
3. Add an AVX2 or NEON kernel only with coefficient- or pixel-exact scalar parity tests.
4. Re-run the 300-frame FFmpeg oracle after each accepted kernel.
5. Record throughput, bytes/op and allocations/op on the same host and fixture.

Current development-host samples are roughly 44-52ms for the BBB baseline, 10.9MB/op and 1,300 allocations/op. They are a regression hint rather than a cross-machine target.

Likely candidates are batched inverse transform/dequant, remaining fractional motion-compensation shapes and deblocking. CABAC is sequential and is not a GPU target.

## Encoder roadmap

Encoder work starts after decoder coverage is broad enough that shared transforms, prediction and bitstream primitives no longer move every week.

The intended phases are:

1. Baseline I/P slices with CAVLC, integer transforms and simple rate control.
2. Main/High support with CABAC, B-frames, 8x8 transforms and weighted prediction.
3. Scalar full-search motion estimation, followed by measured SAD/SATD SIMD.
4. Mode decision and RDO.
5. Optional GPU experiments for batched search or transform work.

The eventual API should accept YUV420 frames and return NAL units without tying callers to the command-line decoder. API shape should be designed when the first end-to-end encoder exists, not beforehand.

## Validation contract

Use a workspace-backed temp directory where `/tmp` is mounted `noexec`:

```bash
export TMPDIR=/workspace/tmp
export GOTMPDIR=/workspace/tmp/go-264
mkdir -p "$GOTMPDIR"

go test ./...
go vet ./...
GOOS=linux GOARCH=arm64 go build ./...
git diff --check
```

The pinned oracle gate is:

```bash
./scripts/bootstrap_fixtures.sh
./scripts/cabac_firstdiv.sh \
  /workspace/tmp/testsrc_cabac_p.h264 \
  /workspace/tmp/go264-cabac-firstdiv

GO264_FFMPEG_REGRESSION=1 \
GO264_FFMPEG_BIN=/workspace/tmp/ffmpeg-7.1.3/ffmpeg \
GO264_BBB_FIXTURE=/workspace/tmp/bbb_annexb.h264 \
go test ./cmd/decode264 -run TestFFmpegReferenceParityBBB -count=1 -v
```

A decoder change is complete when the focused regression passes, the full Go gates are green, the arm64 build succeeds and the FFmpeg oracle remains exact. Performance changes additionally need before/after benchmark data; generated entropy-table changes need a clean `go generate ./entropy/cabac ./entropy/cavlc` diff.
